# Дневной цикл

## Стандартная задача — от и до

```mermaid
sequenceDiagram
    участник Ты
    участник Shell
    участник Agent

    Ты->>Ты: git status (что в проекте)
    Ты->>Shell: opencode
    Ты->>Agent: /analyze
    Agent-->>Ты: Резюме проекта
    Ты->>Agent: /plan <задача>
    Agent->>Agent: создаёт docs/plans/<slug>.md
    Agent-->>Ты: ссылка на план + риски
    Ты->>Ты: читаешь план
    Ты->>Ты: реализуешь вручную (по шагам)
    Ты->>Agent: /review
    Agent-->>Ты: APPROVE / REQUEST CHANGES / BLOCK
    Ты->>Ты: git status / git diff
    Ты->>Ты: git commit (вручную)
```

## Терминал

```bash
# 1. До работы
cd ~/code/my-workspace
git status

# 2. Открыть шелл
opencode

# 3. Внутри шелла
/analyze
/plan Добавить ежедневный backup для n8n

# 4. План создан, читаешь docs/plans/<slug>.md
# 5. Реализуешь шаги из плана вручную
# 6. После изменений
/review

# 7. Если /review дал APPROVE — выходишь из шелла, проверяешь
git status
git diff

# 8. Коммитишь вручную
git add data/inventory.md automations/n8n/backup-daily.md
git commit -m "Add n8n daily backup with verified restore"
```

## Когда что использовать

| Ситуация | Команда |
|---|---|
| Открыл новый проект | `/analyze` |
| Задача > 3 файлов | `/plan` сначала |
| Задача с данными / БД | `/plan` + data-operator |
| Внешний материал | `/intake` |
| Новый data asset | `/inventory` |
| Перед коммитом | `/review` |

## Маленькие задачи

Если задача мелкая (1-2 файла, нет данных, нет рисков):

```
opencode
> Помоги добавить заголовок в docs/overview.md
```

Можно пропустить `/plan`. Но `/review` перед коммитом — всегда полезно.

## Что НЕ делать

❌ Коммитить сразу после агента, без проверки
❌ Игнорировать `git status` — может быть посторонний diff
❌ Запускать `git push` через агента — пуш всегда вручную
❌ Делать `/fix` или подобные — таких команд у нас нет, реализация вручную
