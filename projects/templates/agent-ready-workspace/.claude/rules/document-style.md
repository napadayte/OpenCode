# Document Style Rules

------------------------------------------------------------

## Purpose

Documents must be easy to scan, continue, and verify.

This workspace values clear structure over clever writing.

------------------------------------------------------------

## Separator Rule

Use this separator between major sections:

------------------------------------------------------------

Do not replace it with short dashes or decorative variants.

------------------------------------------------------------

## Writing Style

Use:

- clear headings
- short paragraphs
- direct wording
- explicit paths
- explicit commands
- explicit risks
- explicit next steps

Avoid:

- vague summaries
- motivational filler
- hidden assumptions
- unexplained acronyms
- long dense paragraphs
- decorative complexity

------------------------------------------------------------

## Required Sections For Plans

Plans should include:

- goal
- affected files
- affected data
- environment
- steps
- risks
- verification
- rollback
- approval needed

------------------------------------------------------------

## Required Sections For Reports

Reports should include:

- changed files
- what changed
- why changed
- verification
- risks
- next step

------------------------------------------------------------

## Command Formatting

Commands should be copy-paste safe.

Prefer fenced bash blocks in normal documentation.

When commands are environment-specific, label them:

- WSL
- PowerShell
- macOS
- Docker
- Conda

------------------------------------------------------------

## Placeholder Rule

Placeholders are allowed only when intentional.

Unexplained placeholders should not remain in final template files.

------------------------------------------------------------

## Continuation Rule

Every important document should make it obvious what to do next.

End with one of:

- next step
- open questions
- verification
- decision needed
- owner

------------------------------------------------------------
